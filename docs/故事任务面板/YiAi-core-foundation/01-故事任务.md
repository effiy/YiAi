## 故事 1: 配置管理系统

**作为** 开发者
**我想要** 从 YAML 文件读取所有运行时配置，扁平化为 Pydantic Settings 字段
**以便** 单一配置来源、环境变量覆盖、类型安全

### 验收标准

- Given `config.yaml` 存在，When 应用启动，Then YAML 嵌套 key 被扁平化（`server.host` → `server_host`）注入 Settings
- Given 环境变量 `SERVER_HOST=0.0.0.0`，When 创建 Settings 实例，Then 环境变量优先于 YAML
- Given `config.yaml` 缺失，When 应用启动，Then 使用 Pydantic Field 默认值，不崩溃
- Given 配置包含列表字段（如 `cors_origins`），When 值为逗号分隔字符串，Then 正确解析为 List[str]

### 边界

- 文件: `src/core/config.py`(205行)
- 依赖: pydantic-settings, PyYAML
- 上游: `config.yaml`
- 下游: 所有模块通过 `settings` 单例访问

## 故事 2: MongoDB 数据库访问层

**作为** 开发者
**我想要** MongoDB 单例管理连接池、索引和 CRUD 包装
**以便** 统一的数据库访问、自动时间戳、连接生命周期管理

### 验收标准

## 故事 3: X-Token 认证中间件

**作为** API 消费者
**我想要** 通过 X-Token 请求头认证
**以便** 受保护的端点只接受合法请求

### 验收标准

- Given `middleware_auth_enabled=true`，When 请求缺少有效 X-Token，Then 返回 401 UNAUTHORIZED
- Given OPTIONS 预检请求，When 进入中间件，Then 跳过认证直接放行
- Given 白名单路径 `/read-file`，When 请求该路径，Then 跳过认证
- Given `middleware_auth_enabled=false`，When 任何请求，Then 跳过认证

### 边界

- 文件: `src/core/middleware.py`(112行)
- 上游: `settings.middleware_auth_enabled`, `settings.auth_token`
- 下游: 所有受保护路由
- 白名单: `/write-file`, `/read-file`, `/delete-file`, `/upload`, `/static/*`

## 故事 4: 统一响应包装

**作为** 前端开发者
**我想要** 统一的 `{code, message, data}` 响应格式
**以便** 所有 API 返回一致的结构

### 验收标准

- Given 成功响应，When 调用 `success(data)`，Then 返回 `{code:0, message:"success", data:...}`
- Given 业务异常，When 调用 `fail(error, message)`，Then 返回对应 ErrorCode 的 HTTP 状态码和 business code
- Given 分页数据，When 调用 `success(data, pagination=...)`，Then 响应包含 pagination 字段

### 边界

- 文件: `src/core/response.py`(71行)
- 上游: `core/error_codes.py` (ErrorCode 枚举)
- 下游: 所有路由

## 故事 5: 全局异常处理链

**作为** 开发者
**我想要** 四层异常处理器（验证→业务→HTTP→通用）
**以便** 所有异常都转换为标准错误响应

### 验收标准

- Given Pydantic 验证失败，When 请求参数不合法，Then 返回 400 INVALID_PARAMS 含字段级错误详情
- Given 业务逻辑抛出 `BusinessException(ErrorCode.XXX)`，When 被捕获，Then 返回对应业务错误码和消息
- Given FastAPI 抛出 `HTTPException`，When 被捕获，Then 映射为自定义 ErrorCode
- Given 未捕获异常，When 发生，Then 返回 500 SERVER_ERROR 不泄露内部细节

### 边界

- 文件: `src/core/exception_handler.py`(68行), `src/core/exceptions.py`(19行), `src/core/error_codes.py`(57行)
- 注册点: `src/main.py:register_exception_handlers()`
- 下游: 全应用

## 故事 6: 应用生命周期与工厂

**作为** 运维者
**我想要** `create_app()` 工厂函数管理启动/关闭流程
**以便** 数据库初始化、RSS系统、中间件按正确顺序启动和关闭

### 验收标准

- Given `create_app()` 调用，When 应用启动，Then 按序注册: CORS → Observer → Routes → Auth Middleware → Static
- Given 应用关闭，When lifespan shutdown 触发，Then 关闭 RSS 调度器、关闭数据库连接
- Given `init_db=False`，When 应用启动，Then 跳过数据库初始化（测试模式）

### 边界

- 文件: `src/main.py`(171行)
- 中间件链: CORS → Observer(Sampler→Throttle) → Auth → Route
- 生命周期: init_db → init_rss → yield → shutdown_rss → close_db
