# Observer 可靠性子系统 — 安全审计

## 审计范围

- `src/core/observer/throttle.py` — 限流中间件
- `src/core/observer/sampler.py` — 尾部采样
- `src/core/observer/sandbox.py` — 沙箱执行
- `src/core/observer/lazy_start.py` — 懒启动
- `src/core/observer/guard.py` — 重入守卫

## 安全发现

### [P1] 限流绕过 — X-Forwarded-For 头伪造

- **发现**：限流基于客户端 IP，若服务在反向代理后且未正确获取真实 IP，攻击者可伪造 X-Forwarded-For 绕过限流
- **风险**：限流形同虚设，可发起大量请求
- **建议**：确认获取真实 IP 的逻辑考虑了代理头，必要时信任代理链

### [P2] 限流内存耗尽

- **发现**：限流数据存储在内存字典中，无自动清理僵尸 IP 的机制
- **风险**：攻击者使用大量不同 IP 发起请求，耗尽服务内存
- **建议**：增加 IP 条目上限，超过上限时使用 LRU 淘汰

### [P1] 沙箱绕过风险

- **发现**：沙箱通过 monkey-patch `builtins.open` 实现，若被调用代码使用 `os.open`、`io.open`、`pathlib.Path.open` 等其他方式访问文件系统，沙箱可能被绕过
- **风险**：恶意模块可越权读写文件
- **建议**：扩展 monkey-patch 覆盖更多文件访问路径；考虑使用 OS-level 沙箱（如 seccomp、容器）

### [P2] 沙箱默认关闭

- **发现**：`observer_sandbox_enabled` 默认为 `False`
- **风险**：生产环境若未显式开启，模块执行无文件系统/网络限制
- **建议**：生产部署检查清单中确认沙箱状态

### [P2] Observer 健康检查信息泄露

- **发现**：`/health/observer` 接口暴露限流活跃 IP 数量、采样缓冲区大小、沙箱违规总数等运行时信息
- **风险**：攻击者可利用这些信息判断限流状态、寻找绕过时机
- **建议**：健康检查接口加入认证，或减少暴露的细节信息

## 合规检查

| 检查项 | 状态 | 说明 |
|--------|------|------|
| DoS 防护 | 已实施 | 基于 IP 的请求限流 |
| 文件系统访问控制 | 已实施 | 沙箱白名单（默认关闭） |
| 网络访问控制 | 已实施 | 沙箱白名单（默认关闭） |
| 递归深度限制 | 已实施 | ReentrancyGuard max_depth=3 |
| 运行时监控 | 已实施 | 尾部采样 + 慢请求检测 |
| 安全默认配置 | 部分 | 沙箱默认关闭 |

## 审计结论

Observer 子系统提供了多层防护能力。限流、采样、重入守卫是有效的可靠性保障。沙箱是最强的安全控制但默认关闭且 monkey-patch 方式存在绕过风险，建议生产环境开启并持续加固。
