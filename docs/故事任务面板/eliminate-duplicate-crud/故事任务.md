# 故事任务: 消除重复 CRUD 服务

## §1 Story

| 维度 | 内容 |
|------|------|
| 角色 | 后端维护者 |
| 场景 | 发现 `data_service.py` 和 `mongo_store.py` 包含~750行重复的 MongoDB CRUD 逻辑，且两者均未被任何代码引用 |
| 价值 | 消除死代码减少维护负担，消除混淆（两个文件实现相同功能但接口不同） |
| 范围 | `src/services/database/` |
| 依赖 | 无上游依赖；无下游调用方（两文件均为孤立死代码） |
| 变更级别 | T3（删除文件属范围变更） |

**判定：两文件均未被项目内任何模块导入或调用。** grep 全项目确认 `data_service`、`mongo_store`、`MongoDBService` 仅出现在自身文件名中，无任何 import 或调用引用。

## §3 Design

### 影响链分析

```
data_service.py (461行)
  ├── imports: core.database.db, core.config.settings, core.utils
  ├── callers: NONE (全项目零引用)
  └── action: DELETE

mongo_store.py (535行)
  ├── imports: core.database.db, core.config.settings
  ├── defines: MongoDBService class (18 methods)
  ├── callers: NONE (全项目零引用)
  └── action: DELETE

services/database/ 目录
  ├── 包含: data_service.py, mongo_store.py
  ├── 删除后剩余: 空目录
  └── action: DELETE directory
```

### 安全约束

| 约束 | 验证方法 |
|------|---------|
| 无调用方受影响 | `grep -rn` 全项目确认零 import/调用 |
| 服务启动不受影响 | `python -c "from src.main import create_app"` 导入检查 |
| 不引入 import 错误 | 两文件只有内向 import（`from core.database` 等），删除不破坏其他模块依赖链 |

### 风险

- **风险低**：两文件为零引用死代码。唯一风险是动态/反射调用，但项目未使用任何动态导入机制。
