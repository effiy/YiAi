# 项目总览 — 安全审计

## 审计范围

项目总览作为系统全景知识基线，审计范围覆盖：
- 信任边界识别（外部请求 → Nginx → FastAPI → MongoDB/OSS/Ollama）
- 认证链路完整性（X-Token 中间件）
- 纵深防御层次（CORS → 限流 → 认证 → 路径安全 → 沙箱执行）
- 配置安全（敏感配置的环境变量覆盖）
- 降级安全（各降级路径是否引入新攻击面）

## 安全发现

### [P0] 信任边界清晰

- **发现**：系统存在 4 条明确信任边界：Nginx ↔ FastAPI、FastAPI ↔ MongoDB、FastAPI ↔ OSS、FastAPI ↔ Ollama
- **验证**：每条边界均有对应安全控制（认证、网络隔离、沙箱执行）
- **结论**：信任边界划分合理，纵深防御层次完整

### [P0] 认证链路位置正确

- **发现**：X-Token 认证中间件位于 Observer 限流/采样之后、Route Handler 之前
- **验证**：`src/core/middleware.py` 中的 `header_verification_middleware` 在 `src/main.py` 中注册
- **结论**：认证在业务逻辑处理前执行，位置安全

### [P1] 配置敏感信息保护

- **发现**：`config.yaml` 中的 OSS access key/secret、MongoDB 连接字符串等敏感配置依赖环境变量覆盖
- **风险**：默认 YAML 中若存在硬编码密钥，可能被提交到版本控制
- **建议**：确认 config.yaml 中敏感字段均通过 `${ENV_VAR}` 引用，不包含硬编码值

### [P1] 降级路径安全

- **发现**：OSS → 本地存储降级时，文件安全控制是否一致？
- **风险**：OSS 端可能有服务端加密、访问控制等，降级到本地后这些控制丢失
- **建议**：确认本地存储路径同样受路径遍历防护保护，且访问权限正确配置

### [P2] Observer 组件缺失时的安全

- **发现**：Observer 组件（限流/采样/沙箱/守卫）不可用时仅打印 warning 并跳过注册
- **风险**：限流组件缺失时服务完全暴露，可能被 DDoS
- **建议**：考虑在限流组件不可用时进入"安全模式"（拒绝所有请求或仅允许白名单 IP）

### [P2] Token 缺失降级

- **发现**：Token 缺失时降级跳过认证
- **风险**：若部署时忘记配置 Token，服务将无认证运行
- **建议**：生产环境应强制要求 Token，仅在开发环境允许降级

## 合规检查

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 信任边界识别 | 通过 | 4 条边界明确，各有安全控制 |
| 认证机制 | 通过 | X-Token 头验证，可配置开关 |
| 纵深防御 | 通过 | CORS → 限流 → 认证 → 路径安全 → 沙箱执行 |
| 最小权限 | 待验证 | 需确认 MongoDB 用户权限和 OSS RAM 策略 |
| 配置安全 | 待验证 | 需确认 config.yaml 无硬编码密钥 |
| 降级安全 | 部分 | OSS 降级和 Token 降级需加固 |

## 审计结论

项目总览反映的 YiAi 安全架构整体合理。信任边界清晰，纵深防御层次完整（CORS → 限流 → 认证 → 路径安全 → 沙箱执行）。主要改进点：需验证配置文件中无硬编码密钥，OSS 降级到本地存储时需确保安全控制一致性，Token 缺失降级和 Observer 组件缺失时需考虑更严格的 fail-safe 策略。
