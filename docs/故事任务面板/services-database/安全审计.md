# 数据库服务层 — 安全审计

## 审计范围

- `src/services/database/data_service.py` — 通用数据服务
- `src/services/database/mongo_store.py` — MongoDB 服务封装

## 安全发现

### [P0] NoSQL 注入 — 正则表达式注入

- **发现**：`_handle_string_search_filter()` 虽然使用了 `re.escape(value)` 转义特殊字符，但在多词搜索场景中，用户输入的搜索词被直接拼入正则表达式
- **验证**：代码中使用了 `re.escape(term)` 进行转义，当前实现是安全的
- **结论**：正则注入风险已通过 `re.escape` 缓解，但需确保所有路径都使用了转义

### [P0] NoSQL 注入 — $where / $regex 操作符注入

- **发现**：`filter` 参数直接合并到 MongoDB 查询条件中，如果用户传入 `{"$where": "..."}` 可能造成注入
- **风险**：攻击者可注入 MongoDB 操作符（$where、$regex 等）执行任意 JavaScript 或进行拒绝服务攻击
- **建议**：对 filter 中的 key 进行检查，拒绝以 `$` 开头的 key（除非是已知安全操作符如 `$or`、`$in`）

### [P1] 集合名注入

- **发现**：`collection_name` 参数直接用于 `db[collection_name]`
- **风险**：如果攻击者可控制集合名，可能访问未授权的集合（如 system.profile）
- **建议**：增加集合名白名单校验

### [P2] 分页参数滥用

- **发现**：pageSize 最大 8000，但无单次查询超时限制
- **风险**：结合复杂的正则查询，可能导致慢查询影响数据库性能
- **建议**：增加查询超时限制（maxTimeMS）

### [P2] 数据泄露 — sessions 排除逻辑

- **发现**：sessions 集合自动排除 pageContent，但其他敏感字段（如 messages、tokens）仍会返回
- **风险**：可能泄露 AI 对话历史等敏感数据
- **建议**：根据调用上下文决定是否排除敏感字段

## 合规检查

| 检查项 | 状态 | 说明 |
|--------|------|------|
| NoSQL 注入防护 | 部分 | regex escape 已实施，操作符注入未防护 |
| 集合名白名单 | 缺失 | 任意集合名可访问 |
| 查询资源限制 | 部分 | 有分页上限，无超时限制 |
| 敏感字段保护 | 部分 | sessions 排除 pageContent |
| 输入转义 | 通过 | re.escape 用于用户输入 |

## 审计结论

数据库服务层存在 NoSQL 操作符注入风险（P0），需立即增加 filter key 的 `$` 前缀检查。集合名白名单也是重要的安全加固点。regex escape 使用正确，值得肯定。
