## 功能介绍

项目初始化是 YiAi 工程化建设的第一步，旨在通过标准化文档体系建立可维护、可协作的项目基础。它不是一个运行时功能，而是一套知识资产的生成和管理能力。

**核心特性**
- 🎯 **自动生成**：通过 `/generate-document init` 命令自动扫描仓库并生成文档
- ⚡ **事实驱动**：所有内容来自真实代码和配置，禁止虚构
- 🔧 **可重复执行**：支持 re-init 刷新事实，保留人工约定

**适用人群**
- 新加入项目的开发者
- 负责项目维护和技术决策的工程师
- 需要了解项目架构和约定的协作者

## 快速开始

### 前置条件

- [ ] 已安装 Claude Code
- [ ] 已在 YiAi 项目目录下
- [ ] 有权限读取仓库文件

### 30 秒上手

1. **触发初始化**
   ```bash
   /generate-document init
   ```

2. **等待生成完成**
   - Skill 会自动扫描仓库、检索规范、生成文档、执行审查

3. **查看生成的文档**
   ```bash
   ls docs/
   ls docs/项目初始化/
   ```

4. **阅读入口文档**
   - 先看 `README.md` 了解项目
   - 再看 `CLAUDE.md` 了解开发规范

## 操作场景

### 场景一：新成员首次了解项目

**适用情况**：刚 clone 了 YiAi 仓库，需要快速上手。

**操作步骤**：
1. 打开 `README.md`，阅读【项目概述】和【技术栈】
2. 打开 `CLAUDE.md`，阅读【代码风格与规范】和【开发工作流】
3. 打开 `docs/architecture.md`，查看目录组织和放置规则
4. 根据要开发的功能定位到对应的 `src/services/` 或 `src/api/routes/` 目录

**预期结果**：10 分钟内理解项目结构，能够定位相关代码。

**注意事项**：
- ✅ 文档中的命令可以直接复制执行
- ✅ 目录结构描述与实际目录一致
- ❌ 不要跳过 `CLAUDE.md`，其中包含重要的编码规范

### 场景二：排查 API 认证问题

**适用情况**：调用 API 时收到 401 Unauthorized。

**操作步骤**：
1. 查阅 `docs/auth.md`，了解认证流程和白名单规则
2. 检查 `config.yaml` 中的 `middleware.auth_enabled`
3. 检查环境变量 `API_X_TOKEN` 或配置中的 `middleware.auth_token`
4. 确认请求路径是否在白名单中（`/write-file`, `/read-file`, `/delete-file`, `/upload`, `/static/*`）

**预期结果**：定位认证失败原因并修复。

**注意事项**：
- ✅ 生产环境建议始终启用认证
- ✅ 令牌优先级：环境变量 > 配置文件
- ❌ 不要将生产令牌提交到 Git

### 场景三：生产环境部署

**适用情况**：将 YiAi 部署到生产服务器。

**操作步骤**：
1. 查阅 `docs/devops.md`，确认环境要求
2. 安装依赖：`pip install -r requirements.txt`
3. 配置 `config.yaml`：禁用重载、启用认证、配置有效令牌
4. 启动服务：`uvicorn src.main:app --host 0.0.0.0 --port 8000`
5. 查阅 `docs/security.md`，确认安全自检项

**预期结果**：完成安全加固的生产部署。

**注意事项**：
- ✅ 生产环境必须启用 `middleware.auth_enabled`
- ✅ 使用 `API_X_TOKEN` 环境变量配置令牌
- ❌ 不要在生产环境启用 `server.reload`

### 场景四：文档与代码同步

**适用情况**：代码发生重大变更后，需要更新文档。

**操作步骤**：
1. 再次执行 `/generate-document init`
2. Skill 会自动检测已有文件，按 re-init 策略更新
3. 审查变更 diff，确认事实类内容已刷新
4. 确认人工补充的内容未被覆盖

**预期结果**：文档与代码事实保持一致。

**注意事项**：
- ✅ re-init 会刷新技术栈、目录结构、命令等事实
- ✅ 人工补充的约定默认保留
- ❌ 如发生冲突，Skill 会标注 `> 待确认`，需要人工审查

## 常见问题

### 💡 基础问题

**Q1：项目初始化会修改我的代码吗？**
> 不会。项目初始化只生成文档，绝对不修改任何代码文件。

**Q2：re-init 会覆盖我手动添加的内容吗？**
> 默认保留人工补充的内容。如果与代码事实冲突，会标注 `> 待确认` 提示你审查。你也可以使用 `<!-- AUTO-GENERATED:BEGIN/END -->` 哨兵块包裹希望自动更新的段落。

**Q3：文档中的路径和命令都是真实的吗？**
> 是的。所有内容来自代码扫描，不确定的内容会标注 `> 待补充（原因：…）`，禁止虚构。

### ⚙️ 高级问题

**Q4：如何添加新的文档类型或章节？**
> 修改 `.claude/skills/generate-document/rules/` 下的规范文件，下次执行 init 时会按新规范生成。

**Q5：能否只更新某一个文档？**
> 可以针对具体功能使用 `/generate-document <功能名>-<描述>` 生成该功能的文档集。init 命令负责项目级基础文件。

### 🔧 故障排查

**Q6：init 执行后某些链接无效？**
> 检查相对路径是否正确。如果目标文件不存在，文档中会标注 "待补充" 而非创建无效链接。

**Q7：Mermaid 图表在某些平台无法渲染？**
> Mermaid 语法已通过语法层审查。如平台不支持，可复制图表代码到 [Mermaid Live Editor](https://mermaid.live) 查看。

## 技巧与提示

### 💡 实用技巧

1. **快速定位**：使用 `grep -r "关键词" docs/` 在文档中搜索
2. **对比变更**：re-init 后使用 `git diff` 查看文档更新内容
3. **保持同步**：建议每月执行一次 re-init，或在重大架构变更后执行

### 📚 最佳实践

1. **文档先行**：在开发新功能前，先使用 `/generate-document <功能名>` 生成文档集，明确需求后再编码
2. **冲突审查**：re-init 后务必审查 `> 待确认` 标注，避免事实与约定冲突
3. **哨兵块使用**：对于希望自动刷新的段落，用 `<!-- AUTO-GENERATED:BEGIN/END -->` 包裹
